================================================================================
AURA EVENTS - SYSTEM FUNCTIONAL & SPECIFIC REQUIREMENTS
================================================================================

3. Specific Project Requirements
Aura Events is constructed around robust, decoupled, and micro-transaction-based interfaces. The specifications below define the system-level inputs, processing rules, data states, performance thresholds, and security constraints governing the client application (React) and the API server (Laravel).

3.1 System External Interfaces

3.1.1 User Interfaces (UI)
- Editorial Theme: The user interface implements the "Celestial Union" luxury theme, characterized by warm neutrals (#fdf7ff backgrounds, Elegant Brown body text), Soft Gold accents, and a Glassmorphism elevated card system (backdrop-filter: blur(20px)) with celestially diffused shadows.
- Navigation Framework: A sticky sidebar navigation (C10) containing:
  - Left panel: Corporate logo, project/event selection dropdown, core links (Dashboard, Logs, Analytics, Alerts, API Keys, Settings).
  - Main Panel: Dynamic area updating asynchronously with sub-second response times without full page reloads.
- Responsive Layouts: Fluid CSS grid reflows supporting three major viewports:
  - Mobile (<640px): Sidebar collapses into an overlay drawer; multi-column dashboard tables collapse to single vertical stacks.
  - Tablet (640px-1024px): Sidebar is represented in a compact, icon-only layout.
  - Desktop (>1024px): Sidebar is fully expanded; dashboard columns are displayed in multi-panel grids side-by-side.

3.1.2 Hardware & Server Interfaces
- Hosting Hardware: Single virtual server hosted on an AWS EC2 instance containing:
  - CPU: 8 Virtual CPUs (vCPUs) with x86_64 architecture.
  - RAM: 32 GB of system Memory.
  - Storage: 40 GiB AWS EBS (Elastic Block Store) SSD volume, storing the persistent operating files, Docker images, and database tables.
- Client hardware: Any standard user device (Desktop, Tablet, Mobile) running a modern, HTML5-compliant web browser.

3.1.3 Software Interfaces
- Operating Environment: Linux environment running inside the AWS EC2 instance, orchestrating containerized workloads in a lightweight Kubernetes (K3s) cluster with an Nginx Ingress Controller routing external traffic.
- Database: MySQL 8.0 Community Server running inside a K3s StatefulSet container, with persistent volumes mapped directly to the host's 40 GiB EC2 EBS volume.
- Server-Side Framework: Laravel 12 API running inside a dedicated K3s pod using an Alpine-PHP 8.2 container.
- Client-Side Framework: React 18+ Single Page Application built via Vite, compiled, and served via an Nginx container inside a K3s pod.
- Payment Service Gateway: Razorpay Checkout API and real-time webhook handlers.
- Dispatch Relay: SMTP-based Mail transfer service.

3.1.4 Communications Interfaces
- HTTPS Protocol: All data exchanges between the browser client and the Laravel API are encrypted using TLS 1.3 over port 443.
- JSON Serialization: REST API routes exclusively consume and produce application/json payloads.
- Razorpay Webhooks: Outbound secure HTTPS callbacks POSTed directly from Razorpay servers to `/api/payments/webhook` on the EC2 host.
- SMTP Transport: Secure mail transfer routing via ports 587 (TLS) or 465 (SSL).

3.2 Functional Requirements

3.2.1 User Authentication & Access Controls
3.2.1.1 Introduction:
  System validates registrations, log ins, handles session tokens, and grants role permissions (Planners, Vendors, and Admins).
3.2.1.2 Inputs:
  - Registration: `name` (string), `email` (string, unique), `password` (string, min 8 chars), `password_confirmation` (string), `role` (enum: 'planner', 'vendor').
  - Login: `email` (string), `password` (string).
3.2.1.3 Processing:
  - Server validates parameters. Hashes passwords using the Bcrypt algorithm.
  - Registers the user in the `users` table. If the role is 'vendor', automatically sets up a pending vendor profile inside the database.
  - Generates API access tokens via Laravel Sanctum upon verification.
3.2.1.4 Outputs:
  - Response: A JSON payload containing the user profile object, active `role`, `is_active` status, and a Bearer `token`.
3.2.1.5 Error Handling:
  - Validation failures (e.g., duplicate email): Returns 422 Unprocessable Entity with error fields in a JSON format.
  - Credentials failure: Returns 401 Unauthorized with the message "Invalid credentials".

3.2.2 Event & Workspace Management
3.2.2.1 Introduction:
  Allows planners to maintain detailed event records, checklists (tasks), and wedding timelines.
3.2.2.2 Inputs:
  - Event Creation/Edit: `title` (string), `description` (text), `event_type` (string), `venue` (string), `event_date` (date), `duration` (integer hours), `guest_count` (integer), `budget` (decimal), `cover_image` (file/url).
  - Task: `title` (string), `due_date` (date), `status` (enum: 'pending', 'completed').
  - Timeline Item: `title` (string), `time` (time), `activity` (string).
3.2.2.3 Processing:
  - Automatically maps events to the active planner's `user_id`.
  - Saves corresponding records in `events`, `tasks`, and `timelines` tables.
  - Dynamically calculates the event's `status` (upcoming, Live/Happening, completed, cancelled) using Carbon time validations relative to the event date and duration.
3.2.2.4 Outputs:
  - Response: A JSON payload containing the created event record, dynamically calculated status, associated sub-entities, and updated dashboard numbers.
3.2.2.5 Error Handling:
  - Field validation errors: Returns 422 Unprocessable Entity.
  - Cross-user edit attempts: Returns 403 Forbidden.

3.2.3 Guest Directory & Online RSVP Loop
3.2.3.1 Introduction:
  Planners compile guest listings, assign bride/groom seating sides, and auto-dispatch invitations. Guests confirm attendance online using a secure token.
3.2.3.2 Inputs:
  - Guest Addition: `name` (string), `email` (string), `phone` (string), `side` (string: bride/groom/joint), `notes` (text).
  - Guest RSVP Response (Public): `rsvp_status` (enum: 'confirmed', 'declined'), `food_preference` (string), `rsvp_message` (text).
3.2.3.3 Processing:
  - Saves guest details. The backend automatically creates a secure 40-character unique hash (`rsvp_token`).
  - Triggers SMTP mailers containing the personalized URL link: `/public-rsvp/{token}/{action}`.
  - Public submission: System parses the token, updates guest responses (`rsvp_status`, `food_preference`, `rsvp_message`), sets the response timestamp, and triggers dashboard count updates.
3.2.3.4 Outputs:
  - Response: Guest record with generated token, confirmation notifications on dashboard.
3.2.3.5 Error Handling:
  - Invalid Token: Returns 404 Not Found with the message "Invalid or expired RSVP token".
  - Empty response fields: Returns 422 Unprocessable Entity.

3.2.4 Vendor Marketplace & Service Booking
3.2.4.1 Introduction:
  Vendors list their business services, and planners can search, filter, and request bookings for specific wedding events.
3.2.4.2 Inputs:
  - Vendor Profile: `business_name` (string), `category` (string), `description` (text), `starting_price` (decimal), `location` (string), `image_url` (file/url).
  - Booking Request: `event_id` (integer), `vendor_service_id` (integer), `message` (text).
3.2.4.3 Processing:
  - Saves vendor listing details under `vendor_services`.
  - Planner booking: Creates a `service_bookings` record with a status of 'pending'. Sends notifications to the vendor dashboard.
  - Vendor approves/declines booking request, updating the status column.
3.2.4.4 Outputs:
  - Booking payload with active statuses ('pending', 'accepted', 'declined', 'paid').
3.2.4.5 Error Handling:
  - Service unavailable: Returns 400 Bad Request with "Service is currently not accepting bookings".
  - Unauthorized status update: Returns 403 Forbidden.

3.2.5 Financial Budgeting & Razorpay Payments
3.2.5.1 Introduction:
  Integrates payment requests, Razorpay gateway validation, signature checks, automatic spent_amount calculations, and PDF invoice downloads.
3.2.5.2 Inputs:
  - Payment Request (Vendor): `service_booking_id` (integer), `amount` (decimal), `message` (text).
  - Razorpay Verification (Callback): `razorpay_order_id` (string), `razorpay_payment_id` (string), `razorpay_signature` (string).
3.2.5.3 Processing:
  - Creates a payment record, contacts Razorpay API to generate a transaction, and returns the order parameters.
  - Razorpay UI overlay triggers card/UPI inputs.
  - Verification: Backend receives transaction signatures, computes a local HMAC-SHA256 hash using the Razorpay key, and validates it against the returned signature.
  - If verified: updates payment status to `successful`, booking status to `paid`, adds the payment amount to the event's `spent_amount` inside the budget table, and dynamically generates an invoice PDF.
  - Fallback: Webhook listener updates database asynchronously if the client session terminates.
3.2.5.4 Outputs:
  - Verification payload, updated budget analytics, downloadable PDF invoices.
3.2.5.5 Error Handling:
  - Verification failure: Returns 400 Bad Request with "Payment verification failed".
  - Payment failures: Captures failure logs, updates database column to `failed`, and sends alerts.

3.5 System-Wide Non-Functional Requirements

3.5.1 Performance
- API Response Times: 95% of server read/write operations must process under 100 milliseconds due to local MySQL container hosting on the EC2's high-performance 8 vCPUs.
- Page Transition Speeds: Navigation transitions inside the React SPA must load in less than 500 milliseconds.
- Concurrency: The EC2 virtual server (32 GB RAM) must support up to 500 simultaneous active client sessions without performance degradation.
- Memory: PHP-FPM execution memory limit must be set to a minimum of 256MB.

3.5.2 Reliability
- Database Transactions: Database changes (especially payment updates and booking changes) must employ strict database transactions (COMMIT/ROLLBACK) to prevent split-brain records.
- Mail Retry Queue: Failed invitation emails must be pushed to a queue worker container with a retry limit of 3 before marking as failed.
- MTBF: Mean Time Between Failures must exceed 90 days.

3.5.3 Availability
- Uptime SLA: The platform must maintain a minimum uptime of 99.9% (excluding planned server maintenance windows).
- Workload Self-Healing: Active workloads (frontend SPA Nginx pod, Laravel API pod, MySQL pod, and SMTP mail queue pod) are managed under K3s ReplicaSet controllers and self-healing specifications, with active liveness probes automatically restarting container processes on workload failure or communication lockups.
- Persistent Storage Recovery: The MySQL database workload is managed as a StatefulSet, using a 5Gi PersistentVolumeClaim to map the database folder directly to the host's 40 GiB EBS storage, securing data persistence.

3.5.4 Security
- Password Hashing: Password strings must be hashed with BCRYPT (rounds=10) via Laravel hashes.
- Route Protection: API routes must require Sanctum auth middleware except for login, registration, public rsvps, and webhooks.
- Cryptographic Signatures: Razorpay webhook endpoints must check HMAC-SHA256 header signatures using a shared API key.
- Sanitization: All API inputs must be validated using Laravel Form Requests to block SQL injection and Cross-Site Scripting (XSS) attacks.

3.5.5 Maintainability
- Monolithic Decoupling: Backend must follow strict Laravel Service-Repository patterns. Frontend must utilize Zustand store architectures.
- Docker Standardization: Code must use multi-stage Docker builds to keep the final images clean of dev dependencies.
- Code Comments: Public functions must be fully documented using PHPDoc and JSDoc schemas.

3.5.6 Portability
- Container Native: All code must be operating-system-agnostic and fully functional inside standardized Alpine-Linux-based containers (PHP 8.2 and Node 20).
- Deploy Adaptability: The entire stack can be run or migrated seamlessly to any Docker-capable virtual server (Linux, Windows Server, etc.) without environment mismatch.

================================================================================

3.7 Design Constraints
- Decoupled System: The system must remain decoupled, relying on CORS-compliant JSON REST APIs to ensure high scalability.
- Framework Lock: All server features must stay within the standard libraries of Laravel 12. Frontend styles must remain strictly within the Tailwind CSS 4 utility framework.
- Database Schema Integrity: Database schemas must exclusively use Eloquent migrations. Manual SQL schema manipulation is prohibited.
- Payment Gateway: All credit card/UPI processing must go through the Razorpay SDK to preserve PCI-DSS compliance.
- Host Limitations: The 40 GiB EBS storage limit must be respected by pruning logs and utilizing temporary storage paths for transient invoice generations.

================================================================================

3.9 Other Requirements
- Localization: Backend date-times must be tracked in UTC, converting to the planner's browser locale on the frontend.
- PDF Generation: Invoices must generate in PDF format using standard lightweight PHP packages (e.g. dompdf) compatible with Alpine container bases.
