4. Project Analysis Models
To define the data routing, relational mappings, and system boundaries of the Aura Events platform, several structured models are employed. These models translate the application features into concrete processing rules, database schemas, and data pipelines running on the AWS EC2 instance.

The system uses two primary models:
- Entity-Relationship Model (specifying database schemas in a MySQL 8.0 container).
- Level 1 Data Flow Diagram (specifying data transactions and processes).

4.1 Level 1 Data Flow Diagram (DFD)
The Level 1 Data Flow Diagram (DFD) maps the functional architecture of the Aura Events platform. It traces how data travels from external entities (Planners, Vendors, Guests, and Razorpay APIs) through active system processes, interacts with database stores, and returns as dashboard updates.

--------------------------------------------------------------------------------
DFD Data Flows & Interactions
--------------------------------------------------------------------------------

- Flow 1.0 (Authentication):
  - Action: Planner/Vendor registers or submits credentials.
  - Processing: The auth logic compares inputs with secure Bcrypt hashes inside the [D1: Users] table. Generates and returns a Sanctum API token to the client browser on success.
- Flow 2.0 (Workspace Management):
  - Action: Planners create events, add check-lists, or map day-of schedules.
  - Processing: Stores event metadata in [D2: Events], creates checklist items in [D3: Tasks], compiles timelines in [D4: Timelines], and instantiates budget rows in [D5: Budgets]. Calculates active statuses using date comparisons.
- Flow 3.0 (Guest & RSVP Pipeline):
  - Action: Planners add guests, sending invitation emails. Guests reply online.
  - Processing: Creates guest rows in [D6: Guests] containing a 40-character unique token hash. The server sends an invitation email via SMTP. When a guest submits their confirmation, the public route updates rsvp_status, food_preference, and registers timestamps.
- Flow 4.0 (Marketplace Booking):
  - Action: Vendors compile lists. Planners request bookings.
  - Processing: Services are saved in [D7: Vendor Services]. A booking request generates a pending entry in [D8: Service Bookings]. The system routes real-time notifications to the vendor dashboard for approval.
- Flow 5.0 (Razorpay Payments & Budgeting):
  - Action: Vendors request payouts. Planners authorize card/UPI payments.
  - Processing: The system logs transaction details in [D9: Payments] and calls the Razorpay API. Following successful authorization, the webhook or redirect route verifies signature HMAC hashes, marks the booking as paid, adds the transaction total to the spent amount inside [D5: Budgets], and compiles a downloadable PDF invoice.

A. Project Deployment Appendices
The appendices below provide specific technical configurations, environment setup instructions, and deployment standards governing the hosted application.

A.1 Appendix 1: EC2 K3s Cluster & Nginx Ingress Orchestration
The Aura Events platform is hosted on a single, high-performance AWS EC2 instance (8 vCPUs, 32 GB RAM, 40 GiB EBS volume). Instead of raw Docker Compose, the containerized services are managed inside a lightweight Kubernetes (K3s) cluster with native self-healing capabilities, configured as follows:

1. Pod Topology & Services:
   - Nginx Ingress Controller: Acting as the central entry point on host '13.63.139.72.nip.io', routing root traffic ('/') to the frontend SPA service and API requests ('/api') to the Laravel service.
   - Frontend SPA Pod (event-planner-pipeline-frontend:latest): A production-compiled React 18+ application served via a containerized Nginx instance on port 80. Nginx handles client-side routing by rewriting fallback requests to index.html.
   - Backend API Pod (event-planner-pipeline-backend:latest): Houses the Laravel 12 application on port 8000, handling PHP 8.2 execution and MySQL database queries.
   - Queue Worker Pod: A lightweight Alpine PHP deployment running 'php artisan queue:work' in the background to handle SMTP mail queues asynchronously.
   - Database Workload (mysql:8.0 StatefulSet): MySQL is managed as a StatefulSet to guarantee persistent pod identities and stable network DNS resolution ('mysql-service').

2. Volume Mounting & Storage Persistence:
   - Persistent Storage: A 5Gi PersistentVolumeClaim ('mysql-storage') is bound to the MySQL stateful workload, mounting directory '/var/lib/mysql' directly to a persistent local path on the host's 40 GiB EBS storage.
   - Self-Healing: The K3s controller automatically monitors container health via active liveness probes, executing restarts upon workload crash or responsiveness drop.

A.2 Appendix 2: CI/CD Jenkins Pipeline on AWS EC2
Continuous integration and continuous deployment are fully automated using a Jenkins pipeline runner. The pipeline utilizes the cluster's active kubeconfig context ('/etc/rancher/k3s/k3s.yaml') and executes the following stages:
1. Stage 'Checkout': Fetches the latest source code from the Git repository.
2. Stage 'Build Images': Generates fresh Docker image layers using 'docker-compose build'.
3. Stage 'Import Images Into K3s': Saves the built images as tar files and imports them directly into the K3s container runtime registry ('k3s ctr images import').
4. Stage 'Deploy To Kubernetes': Applies all declarative manifest templates using 'kubectl apply -f k8s/'.
5. Stage 'Restart Deployments': Executes rolling updates ('kubectl rollout restart') on the frontend, backend, and queue worker deployments to pull the newly imported image layers without downtime.
6. Stage 'Wait For Backend': Uses 'kubectl rollout status' to verify that the API backend is fully functional before migrating.
7. Stage 'Run Migrations': Runs database updates ('kubectl exec deployment/backend -- php artisan migrate --force') directly within the running container.
8. Stage 'Run Seeders': Populates dynamic records ('kubectl exec deployment/backend -- php artisan db:seed --force') for active vendors and systems.
